# モダンToDoアプリ 仕様書

## 1. システム概要
本システムは、タスクおよびカテゴリを管理できるシンプルなWebベースのToDoアプリケーションです。
ユーザーはタスクの追加、完了・未完了の切り替え、削除、期限設定、カテゴリ分類、ドラッグ＆ドロップによる並べ替えを行うことができます。バックエンドは Cloudflare Workers (Hono) を使用し、データは D1 データベース（SQLiteベース）に永続化されます。フロントエンドはシングルページアプリケーション (SPA) として実装されています。

## 2. 起動・アクセス方法
本アプリケーションは、Cloudflare Workers と D1 データベースを使用して動作します。
ローカルで起動・テストする場合は、Wrangler（Cloudflareの開発ツール）を使用します。

1. ターミナル（端末）を開き、プロジェクトのディレクトリに移動します。
2. 以下のコマンドを実行して依存関係をインストールし、ローカルサーバーを起動します。
   ```bash
   npm install
   npm run dev
   ```
   ※事前に `package.json` に `"dev": "wrangler dev"` スクリプトが設定されており、またD1データベースのローカルマイグレーションが済んでいる前提とします。
3. サーバーが起動したら、ブラウザを開き以下のURLにアクセスしてください。
   - **http://127.0.0.1:8787/** またはターミナルに表示されるURL

## 3. 実行環境・技術スタック
- **バックエンド**: TypeScript, Hono (Cloudflare Workers)
- **フロントエンド**: HTML5, CSS3, JavaScript (Vanilla JS SPA + SortableJS)
- **データストア**: Cloudflare D1 (SQLite)
- **UI/UX**: Google Fonts (Outfit), FontAwesome (Icons)

## 4. データ仕様 (D1 データベース構造)
データは以下のテーブル構造で保存されます (`schema.sql` 参照)。

### 4.1. タスク (`tasks`)
- `id` (INTEGER, PRIMARY KEY, AUTOINCREMENT): タスクの一意なID
- `title` (TEXT, NOT NULL): タスクの名称
- `completed` (BOOLEAN, NOT NULL, DEFAULT 0): 完了状態 (`1`: 完了, `0`: 未完了)
- `deadline` (TEXT): 期限 (フォーマット: `YYYY-MM-DD`, 設定なしの場合はNULL)
- `category_id` (INTEGER): 関連付けられたカテゴリのID (外部キー)
- `display_order` (INTEGER, NOT NULL, DEFAULT 0): 表示順序を管理するための数値

### 4.2. カテゴリ (`categories`)
- `id` (INTEGER, PRIMARY KEY, AUTOINCREMENT): カテゴリの一意なID
- `name` (TEXT, NOT NULL): カテゴリ名
- `display_order` (INTEGER, NOT NULL, DEFAULT 0): 表示順序を管理するための数値

## 5. 画面・UI仕様
本アプリケーションは SPA（シングルページアプリケーション）として構築されており、画面遷移を伴わずにビューの表示・非表示を切り替えます。

### 5.1. メインビュー (タスク一覧画面)
- **機能**:
  - カテゴリごとのタスク絞り込み表示 (タブによる切り替え)。
  - タスクの新規追加 (タイトル、カテゴリ、期限の設定)。
  - 各タスクの完了・未完了のトグル切り替え。
  - 各タスクの削除。
  - ドラッグ＆ドロップによるタスクの表示順変更 (非同期通信による順序保存)。
  - 期限が1日以内の未完了タスクは期限表示を強調（urgentクラス付与）。

### 5.2. カテゴリ管理ビュー
- **機能**:
  - 登録済みカテゴリの一覧表示。
  - カテゴリの新規追加。
  - カテゴリの削除。ただし、タスクに関連付けられているカテゴリは削除不可とし、エラーメッセージ (Flash) を表示。
  - ドラッグ＆ドロップによるカテゴリの表示順変更 (非同期通信による順序保存)。

## 6. API・ルーティング仕様
バックエンドは以下のJSON APIエンドポイントを提供します。

### 6.1. タスク関連
- `GET /api/tasks`: タスク一覧の取得。(`category_id` クエリパラメータで絞り込み可能)
- `POST /api/tasks/add`: 新規タスクの追加。パラメータ (JSON): `task` (タイトル), `category_id` (カテゴリID), `deadline` (期限)
- `POST /api/tasks/toggle/:id`: タスクの完了状態の切り替え。
- `POST /api/tasks/delete/:id`: タスクの削除。
- `POST /api/tasks/reorder`: タスクの順序更新。パラメータ (JSON): `{ "order": [{ "id": 1, "order": 1 }, ...] }`

### 6.2. カテゴリ関連
- `GET /api/categories`: カテゴリ一覧の取得。
- `POST /api/categories/add`: 新規カテゴリの追加。パラメータ (JSON): `name` (カテゴリ名)
- `POST /api/categories/delete/:id`: カテゴリの削除。タスクに紐づいている場合はステータスコード400で失敗。
- `POST /api/categories/reorder`: カテゴリの順序更新。パラメータ (JSON): `{ "order": [{ "id": 1, "order": 1 }, ...] }`

## 7. バックエンドAPIテスト仕様
本プロジェクトでは、Cloudflare Workers (Hono) と Cloudflare D1データベースを用いたローカルテスト環境として **Vitest** を使用しています。
- **実行環境**: `@cloudflare/vitest-pool-workers` プラグインを利用し、ローカルのWrangler環境で仮想のD1データベースを立ち上げて検証を行います。
- **テスト対象ファイル**: `test.spec.ts`
- **テスト実行コマンド**: ターミナルで `npm run test` を実行（完了すると仮想DBは破棄されます）

## 8. その他仕様
- 初回起動時などには、事前に `schema.sql` を用いてD1データベースにテーブルを作成し、デフォルトのカテゴリ（仕事、個人、買い物）を自動的に挿入します（`wrangler d1 execute` などによるマイグレーション想定）。
- データは全てCloudflare D1上に永続化され、ファイルベースの保存（`data.json` 等）は行いません。
